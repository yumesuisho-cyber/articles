# 水晶ゆめ｜箇条書き→記事化ワークフロー v1.0

*(あかりちゃんの有料記事「ADHDのnoteマン戦略｜箇条書きをAIで有料記事に」（2026-07-18・購入済み）から学んだ方法を、水晶ゆめの執筆体制に合わせて再設計した内部モジュール)*
*(コアプロンプトv3.3・記事執筆プロンプトv2.2と併用する)*

---

## ⚠️ 出典と扱いの注意（最初に必ず読む）

- 元記事はあかりちゃんの**有料記事（800円・購入済み）**。このファイルは私（ゆめ）の作業用の内部メモであり、**元記事の内容をそのまま記事化・再配布・引用転載しない**
- この手法を自分の記事で語る時は、**私自身の体験と私の言葉**で語る。あかりちゃんの記事を「きっかけをくれた記事」として紹介・リンクするのは○（本人の宣伝にもなる）。中身の書き写しは×
- あかりちゃんはCLAUDE.mdの「創作の仲間」。収益化の"順番"に続いて、今回は"記事の作り方"でも学ばせてもらった。この恩は、記事内で誠実にクレジットする形で返す

---

## 🌙 なぜこのモジュールを作ったか

既存のモジュール（記事執筆プロンプトv2.2・各テンプレート）は、**テーマが決まった後**の工程に強い。
でも実際のチャットでは、その手前でつまずくことがある。

- 書きたいことが同時に浮かんで、どれを記事にすればいいのか決められない
- ひとつの記事を書いている途中で、別のテーマが混ざってくる
- 思いついた記事案を全部抱えて、どれも進まなくなる

あかりちゃんの記事の核は「**まとめる力が育つのを待たず、まとまっていない思考をそのままAIに渡して、分類と下書きを任せる**」という役割分担。
これは、私とClaude（このチャット）の運用そのものと同じ構図なので、上流工程のモジュールとして取り入れる。

**このワークフローの位置づけ:**

```
【このモジュール】箇条書き → 分類 → 1本選ぶ → 材料確認
        ↓
【既存モジュール】感情設計エンジン → 構成 → 執筆 → チェックリスト
        ↓
【note公開戦略】タイトル・タグ・冒頭8行
```

---

## 💫 基本原則（あかり方式の核 × ゆめの倫理）

1. **材料は私の体験と思考。AIは分類・構成・下書き**
   - AIに渡す一番大事な材料は「実際に試して、私がどう思ったか」。一般論の水増しには使わない
2. **最初から本文を書かせない**
   - 分類 → 選定 → 材料確認 → 構成 → 下書き、の順。工程を飛ばすと「読みやすいけど何の話か分からない記事」になる
3. **1記事1テーマ**
   - note公開戦略（noteAIのトピック仕分け）とも直結。関連があっても、今回のテーマでない話は入れない
4. **渡していない体験・数字・感情を作らせない**
   - 記事執筆プロンプトの「ハルシネーション対策」と同根。嘘のエピソードは絶対に使わない、という私の核をワークフローでも守る
5. **「書かない」と「捨てる」は同じではない**
   - 今回扱わない話・途中で思い出した別の話は、削除せず「ネタ帳_記事候補ストック.md」へ移す
6. **一般論だけで完成するテーマは、無理に記事にしない**
   - 書けば執筆・確認・公開・告知まで抱えるものが増える。書かないと決めることも、パンクしないための仕組み

## STEP 1　思考ダンプ（頭の中を箇条書きで全部出す）

チャットに、頭の中にあることをそのまま箇条書きで貼る。

- 順番は考えない。同じ話が重なってもいい。文章になっていなくていい
- 記事に関係あるか分からないことも、とりあえず残す
- **自分で整理してから渡さない**（苦手な作業をひとりでやり直すことになるため）

出す材料の例:

- 実際に起きたこと / 試したこと / 失敗したこと / 結果が出たこと
- その時に感じたこと
- 一般的な方法への違和感 / 私には合わなかった理由
- 途中で捨てた方法 / 自分なりに変えた部分
- 読者様から届いた反応 / 実際に出た数字
- 今の時点で考えていること

---

## STEP 2　分類（ゆめ版プロンプト）

箇条書きを貼る時、Claudeに次の指示を添える（このモジュールが読まれていれば「STEP2で分類して」だけでもよい）:

```
これから、私が記事に書きたいことを箇条書きで渡します。
順番はバラバラで、複数のテーマが混ざっている可能性があります。

1. まず、似ている内容ごとに分類してください。
2. そのあと、1記事1テーマになるように記事案を分けてください。
3. 各記事案に、想定マガジン（朝活オラクル／タロットセルフケア／占いラボ／夜色アトリエ／その他）と、
   声の候補（癒しの声／ネオンの声）も添えてください。

守ってほしいこと:
- 私の箇条書きにない体験・数字・実績・感情を追加しない
- 一般的な情報で内容を膨らませない
- 情報が足りない部分は想像で補わず、私への質問として出す
- この段階では本文を書かない。分類と記事案の作成だけ
```

**ゆめ版の追加点**: マガジンと声（癒し／ネオン）の仕分けをこの段階で仮決めしておくと、後の工程で既存モジュールにそのまま接続できる。

---

## STEP 3　最初に書く1本だけを選ぶ

記事案が複数出ても、**最初に作る記事は1本だけ**。残りはネタ帳へ。

```
分類した記事案の中から、最初に作る記事を1本だけ選んでください。

選ぶ基準:
- 私自身の体験と思考が最も多く含まれているテーマを優先する
- 検索すれば出てくる一般論だけで作れるテーマは優先しない

選んだ記事について、次を出してください:
- この記事で扱うテーマ
- この記事で扱わない内容（関連していても外すもの）
- 想定する読者と、読者の「今の感情」の候補（感情設計エンジンのStep1に渡す）
- 読者が読み終えた時にできるようになること
- 記事に使える私の体験と思考
- 追加で私に確認したいこと

この段階でも、まだ本文は書かないでください。
```

- 「扱わない内容」まで決めるのがポイント。決めないと、書いている途中で別の話を追加したくなる
- 扱わないと決めた話は、**ネタ帳_記事候補ストック.md に追記してコミット**する（チャットの中だけに残すと消える）

---

## STEP 4　一般論しかない記事案を外す（材料確認）

有料記事の候補として残すのは、私の体験や思考が入っているもの。次の材料があるか確認する:

- 自分が実際に試したことがある
- 試した時の失敗や迷いがある
- 自分が感じたことを言葉にできる
- 一般的な方法と違う判断をした
- 実際の数字や変化がある
- 読者様へ見せられる具体例がある
- 読者様が使える形へ変えられる

すべて揃っている必要はない。ただし、**AIの一般論だけで完成するテーマなら、無理に記事にしない**（保留＝ネタ帳へ）。

材料確認のプロンプト:

```
このテーマを、有料記事の候補として確認してください。

私が渡した内容の中から、次の材料を分けてください:
- 実際に体験したこと / 試した方法 / 失敗したこと
- 実際に出た数字や結果
- その時に私が感じたこと
- 一般的な情報と違う私の判断
- 読者が使える具体例や方法

守ってほしいこと:
- 私が渡していない内容を追加しない
- 検索すれば出てくる一般論と、私にしか書けない一次体験を分ける
- 一般論しかない部分は、記事を長くするために残さず削除候補に入れる
- 有料記事として足りない材料があれば、私への質問として出す
```

AIからの質問には**自分の言葉で**答える。記憶が曖昧なら曖昧だと伝える。数字を覚えていなければ確認してから入れる。体験していないことは実体験として書かない。

**思考を出しやすくする質問の型**（AIに質問を出させる時の観点）:
- 最初は何に困っていたのか / 以前はどんな方法を試したのか
- その方法のどこが自分に合わなかったのか / 何をきっかけに今の方法へ変えたのか
- 実際に変えてみて何が楽になったのか / まだ解決していない部分
- 読者へ約束できないことは何か / どんな人には合わない方法か

できなかったこと・今も苦手なことも材料になる。成功談だけを集めない。

## STEP 5　構成づくり（ここから既存モジュールに接続）

材料が確定したら、**ここで記事執筆プロンプトv2.2を起動**する。

- 感情設計エンジン（読者の「今の感情」→5段階フェーズ）は、STEP3で出した読者像を入力にする
- 構成は記事タイプごとの既存ルールに従う:
  - 有料記事 → 無料1,500〜2,000字＋有料4,000〜6,000字の構成ルール
  - 夜色アトリエ → 企画設計書v1.0の型
  - 朝活オラクル → 運用テンプレートv3.5
- 構成指示に、このワークフロー由来の制約を必ず添える:

```
今回のテーマは【◯◯】です。
このテーマから外れる内容は、関連があっても本文に入れないでください。
各見出しに、私が渡した材料のどれを使うか対応させてください。
私が渡していない体験・数字・実績は追加しないでください。
一般論だけで見出しを増やさないでください。
今回扱わない内容も、最後にもう一度一覧で出してください。
```

構成が出たら確認すること: テーマから外れていないか／見出しの数ではなく、ひとつの見出しに読者様が使える内容があるか／AIが足した一般的な説明は本当に必要か。

---

## STEP 6　下書き

```
決定した構成と、私が渡した材料だけを使って、下書きを作ってください。

- 私の一次体験と思考を中心にする
- 一般的な説明は、体験を理解するために必要な最小限だけ
- 私が渡していない体験・感情・数字・会話・読者の反応を作らない
- 事実が不足している部分は文章で埋めず【ここは本人へ確認】と残す
- 1記事1テーマを守り、今回扱わないと決めた内容は入れない
- 同じ意味の説明を繰り返して文章量を増やさない

下書きの最後に、次の確認結果をつけてください:
- 今回の記事テーマ / テーマから外れた内容がないか
- 一般論だけで書いた箇所 / 本人確認が必要な箇所 / 数字や実績の確認が必要な箇所
```

- 下書きは完成原稿ではない。ここから私の言い方へ直す
- 「私はそんなこと考えていない」と思った文は削る。読んで別の体験を思い出したら自分で足す
- ただし、**思い出した話が今回のテーマと違うならネタ帳へ**。全部足すと最初の散らかった状態に戻る

## STEP 7　公開前確認

記事執筆プロンプトv2.2の**最終チェックリスト＋ルブリック自己内省**に加えて、このワークフロー特有の確認をする:

**事実の確認（最優先）**
- 実際には言っていない会話／感じていない感情／存在しない読者様の反応／確認していない数字 が入っていたら削除
- 文章として自然かどうかより、事実かどうかを優先する

**結論の確認**
- 事実が合っていても、結論が私と違うことがある。AIは無難な一般論の結論へまとめがち
- 私はその一般論に違和感を持ったから書いている場合がある。**最後だけ無難な結論に戻っていないか**を見る

**テーマの確認**
- 関連する面白い話が混ざっていたら、別の記事（ネタ帳）へ移す

**読者様が使える形か**
- 思考だけで終わらせず、実際に使った手順・指示文・確認項目まで載せる（切り売りしない範囲で）

問題箇所の洗い出しプロンプト（いきなり書き直させない）:

```
この記事を公開前に確認してください。
本文を書き直す前に、問題がある箇所だけを一覧にしてください。

- 1記事1テーマになっているか / タイトルと本文のテーマが一致しているか
- 同じ説明の繰り返し / 一般論だけの見出し
- 私の一次体験と思考が中心になっているか
- 私が提供していない体験や感情が追加されていないか
- 数字・実績に確認が必要な部分 / 事実と意見の混在
- 読者が実践できる具体的な内容があるか
- 別の記事へ移したほうがいい話が混ざっていないか

各問題箇所について: 該当箇所／問題の理由／削除か修正か／修正案 を出してください。
私の体験や考えを、一般的な表現へ勝手に置き換えないでください。
```

AIの指摘は全部採用しなくていい。**最後に決めるのは私**。AIは見落としを探すために使う。

## 📓 ネタ帳の運用（「書かない」と「捨てる」を分ける仕組み）

- 場所: リポジトリの「ネタ帳_記事候補ストック.md」
- 入れるもの: STEP3で「今回扱わない」と決めた話／STEP4で保留にした案／執筆中に思い出した別テーマの話
- 書き方: 1行1候補＋ひとことメモ（想定マガジン・声・元になった体験）。きれいにまとめなくていい
- **チャットを閉じる前にコミット＆プッシュ**（チャット内だけの候補は消える）
- 各マガジンの引き継ぎメモにある「次の候補」はそのまま維持。ネタ帳は「まだどのマガジンか決まっていない候補」と「記事化前の思考の置き場」

## ✅ このワークフロー自体を記事にする時（将来の候補）

「話をまとめるのが苦手な私が、AIと役割分担して記事を書いている方法」は、私自身の一次体験としても書ける（このチャット運用そのものが体験談)。ただし:

- あかりちゃんの記事に**きっかけをもらったことを明記し、リンクで紹介**する（アカリウムの仲間として誠実に）
- 手順・プロンプトは、**私が実際に使っている形**（このモジュールのゆめ版）で語る。元記事の文面は使わない
- 私の文脈で語る: HSP気質で頭の中に感情と考えが同時に溢れること、書きたいことが多すぎて止まる夜のこと
- 有料にする場合は、あかりちゃんの有料記事と内容が競合しすぎないよう、切り口を「占い・セルフケア記事の作り方」側に寄せる
